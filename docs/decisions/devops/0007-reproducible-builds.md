# Reproducible and signed release images

## Context and Problem Statement

Nothing lets a third party confirm a published `zebrad` image was built from the tagged source. Two independent properties would establish that: a reproducible build, where anyone rebuilds the same bytes from source, and signed provenance, where the build platform attests who built what.

The two are distinct, and SLSA v1.0 ties neither to the other. Build L3 is about an isolated builder and non-falsifiable provenance; reproducibility was the deferred L4 idea. We want both for release images.

Zebra already builds with `--locked` and `CARGO_INCREMENTAL=0`, pins apt versions, emits `provenance: mode=max` plus an SBOM, and bakes no wall-clock timestamp into the binary. The builds were not yet reproducible, the attestations were unsigned, and released binaries recorded no commit.

## Decision Drivers

- Reuse the existing multi-stage Dockerfile, GitHub Actions, and Docker Build Cloud. Avoid a second build system.
- The artifact is a native binary and its OCI image, so verification rests on the image digest and its attestations, not a consensus blob.
- Reproducibility hardening must be cheap enough to apply to every build; reserve signing for release images.

## Considered Options

1. Harden the existing buildx pipeline: pin every input, remove nondeterminism, and sign with Sigstore.
2. Adopt a hermetic build system such as GNU Guix, as Bitcoin Core, Tor, and Monero do.
3. Sign provenance but leave the build non-reproducible.

## Decision Outcome

Chosen: option 1. Guix gives the strongest guarantee but is a large departure from the current stack and disproportionate for a validator node. Signing alone cannot be independently corroborated. Hardening the existing pipeline reaches both goals with changes a reviewer can read.

Reproducibility applies to every image build, releases included, because the changes are cheap. The build becomes a function of its inputs and nothing else:

- Pin the Rust toolchain to an exact version and digest-pin the rust and debian base images; bump both deliberately.
- Set `SOURCE_DATE_EPOCH` from the commit and rewrite layer timestamps.
- Remap absolute build paths out of the binary.
- Drop the apt/dpkg logs and ldconfig cache, which record wall-clock times in their file contents.

The build stays hermetic by keeping `.git` out of the context and passing the commit as the `SHORT_SHA` build arg. Reading `.git` would let the clone shape, a detached-HEAD tag checkout or a shallow clone, change the output bytes.

Signing and attestation apply on the release path only; the reproducibility measures above apply to every build. Development and dispatch builds are reproducible but unsigned. The `merge` job signs the final multi-arch digest with Cosign and attaches a signed SLSA provenance attestation and a signed SBOM. A verify step then fails the release if any of them do not verify, pinning the signer with `gh attestation verify --signer-workflow`. Release builds also run with `no_cache: true`.

`codegen-units` stays at the cargo default: the binary already reproduces bit-for-bit with thin LTO, so forcing it to `1` would only cost build time.

### Expected Consequences

- Two builds of the same commit produce the same binary. A double-build hash comparison catches regressions from a dependency or base bump.
- The toolchain and base images now move only on deliberate, reviewable bumps.
- Released binaries carry their commit again, and release images are verifiable with `gh attestation verify` or `cosign verify`.
- This reaches signed (L2-grade) provenance, not L3. L3 also needs an isolated builder and provenance generated by a separate workflow the build cannot influence.
- The account-creation date `adduser` writes to `/etc/shadow`, a former drift source, is pinned by `SOURCE_DATE_EPOCH`, which the shadow tools honour. The `zebrad` binary reproduces bit-for-bit.

## More Information

Verify a release image (`<version>` is the release tag):

```sh
gh attestation verify oci://docker.io/zfnd/zebra:<version> \
  --owner ZcashFoundation \
  --signer-workflow ZcashFoundation/zebra/.github/workflows/zfnd-build-docker-image.yml
cosign verify docker.io/zfnd/zebra:<version> \
  --certificate-identity-regexp='^https://github\.com/ZcashFoundation/zebra/\.github/workflows/zfnd-build-docker-image\.yml@' \
  --certificate-oidc-issuer='https://token.actions.githubusercontent.com'
```
